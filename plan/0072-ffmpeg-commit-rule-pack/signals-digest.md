# ANGLE: ffmpeg-ai-policy

SUMMARY: Searched ffmpeg.org developer docs (developer.html live 2026-07-17; doc/developer.texi at master via GitHub mirror and at the corpus-pinned rev a7ffc46c), the ffmpeg-devel archives (pipermail and mail-archive.com, 2025-2026), FFmpeg community/governance page, the FFmpeg/FFmpeg commit history for doc/developer.texi, and 2026 press coverage of the AMD incident. Headline state as of 2026-07-17: FFmpeg has NO adopted AI/LLM contribution policy — no ban, no disclosure requirement, no GA vote or TC decision. A July 2025 RFC ("[RFC] Introducing policies regarding 'AI' contributions", Alexander Strasser, 2025-07-01) proposed creating one and produced disclosure/responsibility proposals (all PROPOSED, none adopted). The operative regime is per-maintainer discretion under ordinary review standards, enforced in practice: a Feb 2026 AMD patch was publicly rejected as unreviewed "AI slop", with the demand of "thorough manual review of all AI-generated code before submission". The only AI text in the official developer docs is new: commit 01a9a98 (Michael Niedermayer, 2026-07-13) added an optional official LLM review bot ("Forgejo Fairy") to the reviews section — the corpus's pinned texi revision a7ffc46c predates this and contains no Fairy/LLM text. Sign-off carries Linux-kernel DCO semantics (the checklist links kernel submitting-patches.rst "Sign your work"), so a human submitter certifies origin of agent-drafted code; one core maintainer (Rémi Denis-Courmont, 2025-11-11) has argued DNN-generated patches are LGPL-incompatible, contested by Niedermayer and not policy. Limitation: code.ffmpeg.org pages (Forgejo_Fairy profile, michaelni/Fairies repo) are Anubis-bot-blocked and could not be fetched; no evidence anywhere suggests a policy is hosted there. Tangential context found via search only (not fetched): the Oct-Nov 2025 Google "CVE slop" dispute concerned AI-generated bug reports, not patches.
