## 1. The five invariants: soundness and completeness

**`DreamIsReadOnly` (251–254) — unsound, and not about read-only-ness.** The expression says "if `saw_finding` then the status is terminal." Two defects. First, it is violated in every reachable run with at least one finding: `RecordFinding` (204–207) sets `saw_finding = true` while `status` is still `dreaming`, and `DreamCompletes` is a separate external stimulus that arrives later. The state between the first finding and the verdict falsifies the disjunction — and that state is quiescent (the detector cascade has settled), so it fails under any invariant-checking granularity. Second, it *accepts* the actually dangerous state: a `Finding` created after a `clean` verdict (nothing guards `RecordFinding` on `dream.status = dreaming`) yields `saw_finding = true, status = clean` — a verdict that lied — and the third disjunct makes the invariant hold there. It rejects the benign mid-scan state and blesses the harmful one. The prose property (247–250: the audit never mutates; all writes come from the MCP surface) is not encoded anywhere and cannot be in the current model: `MemoryWrite` (228–231) has no origin/caller attribution, so "the audit never writes" is a frame condition with no state to check. Fix: add `origin: audit | mcp_tool | consolidation` to `MemoryWrite`, assert `for w in MemoryWrites: w.origin != audit`, and delete or rename the current expression.

**`RepoStoreAppendOnly` (260–263) — sound, but narrower than its name.** It is discharged by `MemoryWriteRefusesRepo` (233–237), and what it encodes is "the MCP tool refuses every repo write" — the repo store is MCP-*read-only*, not append-only. The actual append-only property (an existing entry's text never changes) has no state to range over: the spec has no entry-mutation action; `is_append_only_violation` (60) is an observation input from the analysis pass, not a modeled write. Acceptable at this altitude (the mutation surface is git, out of scope), but the name over-claims what is proved.

**`FixNeverTouchesRepo` (268–271) — sound, leaning on an unstated binding.** It quantifies over `s.target_store`, a *copy* denormalized by `ScopeSuggestion` (193). It holds today only because `ScopeSuggestion` is the sole Suggestion creator and copies faithfully. Nothing asserts `s.target_store = s.finding.entry.store`, so a future Suggestion creator (the consolidate path is the obvious candidate) that mis-copies `target_store` bypasses the repo protection while this invariant stays green. See M3 below.

**`RepoFindingsNeverEdit` (276–279) — the property is right; the spec violates it.** `DetectDanglingLink` (162–166) has no store guard and emits `route: edit` unconditionally. Counterexample in three steps: a `MemoryEntry` with `store.loc = repo`; a `LinkRef` with that entry as `src` and `resolves = false`; the rule creates a Finding with a repo-store entry and `route = edit`. `allium analyse` should surface exactly this trace. The comment at 161 ("editable on either store") asserts the violation as intent, contradicting both the `Route` comment (29–30) and CLAUDE.md section 6. Fix: split by store — per_user → edit, repo → append (a suggested correcting entry with the forward `[[link]]`).

**`EveryFindingHasARoute` (283–286) — vacuous.** `Finding.route` is typed `Route` (76) and `Route` = {edit, append} (36–39), so the disjunction is a tautology the type system already guarantees; it can never fail and does no analytic work. The defect it presumably targets — the `madar` emissions — is a checker-level failure caught before any invariant runs (`madar` resolves to neither a binding nor a `Route` literal). And if `Route` ever legitimately grew a member, this invariant would start rejecting it: it hardcodes the enum's contents a second time. Delete it, or re-aim it at something the type system cannot see (M2).

**Missing invariants:**

- **M1 — `VerdictMatchesFindings`.** Prose line 6: "settles to exactly one terminal verdict: clean (0) or findings (1)" — the exit code is the tool's entire contract, and nothing encodes `d.status = clean implies not d.saw_finding`. It is also not provable today: (a) `saw_finding` is never initialized — `StartDream` (199–202) sets only `status`, so a run may begin with a stale latch and report `findings` on a clean store; (b) `RecordFinding` lacks a dreaming-guard, so a late Finding flips the latch under a settled `clean` verdict. Fix: `StartDream` also ensures `saw_finding = false`; guard `RecordFinding`; add the invariant.
- **M2 — Detector completeness.** Prose lines 3–5 promise both stores are scanned for staleness, contradictions, drift. The rule set silently drops two repo cases: `is_stale_state` and `is_workaround_not_plan` fire only for `per_user` (149–159). A stale repo entry produces no Finding, no route, exit 0 — and no invariant notices, because nothing connects a raised precondition flag to Finding existence. Add per-detector completeness invariants (`for e in MemoryEntries where e.is_stale_state: exists f in Findings where f.entry = e`).
- **M3 — `SuggestionTargetsItsFinding`**: `for s in Suggestions: s.target_store = s.finding.entry.store`. Without it, `FixNeverTouchesRepo` protects a copy, not the store the fix actually hits.
- **M4 — Audit-never-writes.** The real content of `DreamIsReadOnly`; needs `MemoryWrite.origin` first. Until then, "read-only" (6–7, 247) is asserted nowhere in the model.
- **M5 (minor) — acceptance biconditional** `w.accepted iff w.target_store.loc = per_user`. The invariant set encodes only the refuse direction; the accept direction exists only as rule `MemoryWriteAcceptsPerUser` (239–243), so a regression there — or a third `StoreLoc` later — is invisible.

## 2. Paired detectors vs post-creation route rewrite

**Creation-time routing is the right call; the pairing is the weaker of the two creation-time encodings.**

Why creation-time beats a rewrite rule: Allium invariants are state assertions with no temporal operators. A rewrite rule (`when: f: Finding.created, requires: repo and route = edit, ensures: f.route = append`) manufactures a transient state where `RepoFindingsNeverEdit` is false; the invariant would have to be weakened to settled states only — exactly the "eventually correct" shape the language cannot check cleanly, and which the stale comment at 273–274 still half-describes. There is also a rule-ordering race: `ScopeSuggestion` fires on the same `Finding.created` trigger (191–195), and with a rewrite rule in play, whether a Suggestion is scoped before or after the route settles is unspecified. Creation-time routing removes the race class rather than adjudicating it, and keeps every invariant inductive (check each creation rule once) instead of forcing the analyser to reason about interleavings.

The cost — demonstrated in this very spec — is that pairing turns store-coverage into a by-hand exhaustiveness obligation with no checker backstop. Three of six detectors have already drifted: stale-state and workaround-vs-plan lost their repo halves (149–159), and dangling-link (162–166) skipped pairing entirely and broke the invariant. Nothing forces "per detector, the per_user and repo `requires` are jointly exhaustive."

Recommended encoding — same semantics, half the rules, no silent-drop failure mode. Allium ensures blocks compose with `if`/`else`:

```
rule DetectDescriptionBodyDrift {
    when: entry: MemoryEntry.created
    requires: entry.has_description_body_drift
    ensures:
        if entry.store.loc = per_user:
            Finding.created(entry: entry, kind: "description-body-drift", route: edit)
        else:
            Finding.created(entry: entry, kind: "description-body-drift", route: append)
}
```

One `requires` per detector means the store split cannot eat a case; routing stays creation-time; `RepoFindingsNeverEdit` stays inductive. If the pairs are kept instead, add the M2 completeness invariants so a missing half is a caught defect rather than a silent coverage hole. Either way, delete the sentence at 122–123 ("The route emitted here is the detector's *default*; the invariant below constrains the actual verdict") — it documents the rejected rewrite design, and spec comments are contract prose.

## 3. MCP-surface gaps

`MemoryWrite` is two fields (228–231): a store pointer and an accept bit. Everything below is invisible to it.

- **Update vs create — missing.** No operation kind, no target entry. Risk: the end-to-end property the plan exists for — "consolidation applies finding F's suggested edit to entry E" — is unstatable; a consolidation that writes the right store but the wrong entry, or creates a duplicate instead of editing, satisfies every invariant. Deletion is real on the per-user store (consolidation prunes), and deleting an entry that others `[[link]]` to manufactures dangling links — the audit's own defect class; no invariant says a fix does not create findings. Minimum: `op: create | update | delete`, `entry: MemoryEntry?`, and a rule tying an applied Suggestion to the write that discharges it.
- **Idempotency — missing.** No request identity, no at-most-once tie to a Suggestion. Risk: an MCP transport retry or harness re-invocation double-applies; for append-shaped writes that means duplicate entries, which the next dream flags as contradiction/drift — the fixer becomes a finding generator. Cheap fix: `write.suggestion: Suggestion?` plus "at most one accepted write per suggestion."
- **`memory_consolidate` — absent entirely.** It exists only in the comment at 225–227 ("calls memory_write on the per-user store only") — the strongest safety claim on the mutation side, and it is a frame condition with no state behind it. Risk: nothing constrains a consolidation to apply only Suggestions with `applies = true`; a consolidation writing arbitrary content, or "fixing" a repo-store finding by writing a shadow copy into the per-user store (accepted — it is per_user), is fully invariant-clean. Also unmodeled: whether consolidate re-runs the detectors, and which Dream run's Suggestion set it consumes.
- **Concurrency / atomicity — unmodeled.** (i) TOCTOU verdict staleness: a write lands between the analysis pass populating the precondition booleans (47–51) and `DreamCompletes`; the verdict describes a store that no longer exists. Tolerable for an advisory tool, but say it — a `@guarantee` on `DreamRun` scoping the verdict to a snapshot. (ii) In fix mode, the Suggestion's verbatim text (79) was computed against content a concurrent write may have changed; `MemoryWrite` has no expected-version field, so compare-and-swap is inexpressible and lost updates are unmodeled. (iii) A multi-write consolidation (edit A + append B) has no transaction; partial application can leave A's new `[[link]]` pointing at a B never created.
- **Partial failure — `accepted` conflates authorization with effect.** The per-user store is physically two writes (entry file + `MEMORY.md` index line); a crash between them is index/entry divergence — adjacent to the drift class the audit detects, and the index-points-at-missing-file case is a dangling reference `LinkRef` cannot even represent (it models entry-to-entry links only, 64–65). Risk: `accepted = true` reported over a half-applied write. Minimum: split `accepted` (policy) from `completed` (durable effect) and hang the guarantees off `completed`.

One deliberate gap worth a line in the spec: no modeled actor can execute an `append`-routed repo suggestion — memory_write refuses the store and `applies` is pinned false. The executor is the operator via a linted git commit, outside the model. State that next to the `Route` enum, or a reader will hunt for the missing repo-append path and conclude the spec is incomplete rather than advisory.

## 4. Reviewer-raised findings

- **F1 — `route: madar` (174, 183) does not exist, and must not.** `madar` resolves to neither a binding nor a `Route` literal, so `allium check` cannot exit 0 — the plan's write-spec gate ("check + analyse exit 0, zero findings") fails before analysis begins. This is not merely a typo for "madr": the non-overlap verdict in gather-data.md explicitly retired it. The enum and its comment (26–35) carry the post-confirmation design; the two rules and the header (line 5: "edit / append / MADR") still carry the pre-confirmation design from the gather-data detector table. The spec is caught mid-migration, self-inconsistent in exactly the direction the non-overlap ruling closes off. Under the confirmed decision: append-only-violation → `append` (repo; the suggested correcting entry — matching CLAUDE.md section 6's own convention that corrections are new entries); room-touching → store-split like the others (per_user → edit, repo → append), naming the superseded room record as *context in the finding text*, never as a route. `DetectRoomTouching` (180–184) also needs the store guard it currently lacks.
- **F2 — the run lifecycle cannot be re-entered, and the graph knows it.** The transition graph (94–98) is authoritative in Allium 3 — rules producing transitions outside it are validation errors — and it has no `-> dreaming` edge. `StartDream` (199–202) carries no `requires`, so it can produce `clean -> dreaming` on a second `DreamStarts` (the surface's `provides` has no when-guard, 105–110): at minimum a checker question, likely a validation error. Combined with `saw_finding` never being initialized or reset, the model supports exactly one run per universe without saying so. Either declare single-shot (guard `DreamStarts`) or add re-run edges plus the `saw_finding = false` reset. Related: `Finding` has no reference to its Dream (73–77), so under any multi-run reading, verdicts bleed across runs through the single given `dream`.
- **F3 — stale identifier in a comment.** Line 121 names "RouteRepoFindingsNeverEdit"; the invariant is `RepoFindingsNeverEdit` (276). A dangling name reference — in the spec whose flagship detector is dangling links.
- **F4 — `Finding.kind` is a String (75).** The six kind names are convention, not checked; a typo'd kind in a future rule is invisible, and completeness invariants cannot quantify per kind. An open String may be deliberate while detector naming is under cast review (13–15), but once naming settles it should become an enum.
- **F5 — `fix_mode` (93) is never assigned.** As an unbound input, the analyser may flip it mid-run between Finding creations, scoping some suggestions applicable and others not within one run. It is a per-run CLI flag: bind it at `DreamStarts`.
- **F6 — no failure verdict.** `status: dreaming | clean | findings` (91) gives a crashed scan (unreadable store, malformed entry) no modeled outcome, while the header pins the exit contract to 0/1. CLI strings are out of scope, but the verdict machine is in scope; an `error` terminal keeps `findings` from becoming the dumping ground for "crashed."
- **F7 — cross-store links unmodeled.** LinkRef resolution is defined same-store (64–65). If `[[slug]]`s may cross stores — and the per-user tier's value is precisely that it references repo lore — same-store resolution makes every cross-store link a false-positive dangling-link finding. Either record "links are same-store by format definition" as a spine/format decision, or resolve across both stores.

**Priority order:** (1) F1 — fails the gate and contradicts the recorded ruling; (2) the `DetectDanglingLink` store split — an analyser-findable invariant violation; (3) `DreamIsReadOnly` → replace with M4 (needs `MemoryWrite.origin`) and add M1 with the `saw_finding` init/guard fixes; (4) M2 completeness, ideally via the single-rule `if`/`else` encoding; (5) the MCP-surface gaps (op/entry, consolidate as a modeled Suggestion consumer, accepted vs completed); (6) comment hygiene (F3, 122–123, line 5) — cheap, and in a spec, comments are contract prose.
